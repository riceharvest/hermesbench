# Hugging Face dataset candidates for Hermes v0 SFT

Checked on Hugging Face for public datasets close to the current target: compact Hermes-style agent traces for tool choice, repo/terminal work, verification discipline, and high task success per reasoning token.

## Short answer

No public dataset found is exactly our target format (`ACTION ...`, `SCRATCH<=32`, concise verified finals). The closest useful sources are agent trajectory datasets that need conversion/filtering/compression before entering `data/processed/hermes_v0_train.jsonl`.

## Best candidates

### 1. `open-thoughts/OpenThoughts-Agent-v1-SFT`

- Why relevant: terminal/code/software-engineering agent SFT dataset; built for Terminal-Bench/SWE-Bench style tasks.
- Schema: `conversations`, `agent`, `model`, `model_provider`, `task`, run metadata.
- Format observed: assistant emits JSON with `analysis`, `plan`, `commands`, `task_complete`.
- Fit: high for terminal-agent behavior, but too verbose for our ultra-compact target.
- Use: import only successful command/action turns; compress `analysis+plan` into `SCRATCH<=32` or `ACTION terminal`.

### 2. `nvidia/SWE-Zero-openhands-trajectories`

- Why relevant: large OpenHands-style software engineering trajectories; generated by Qwen3-Coder-480B-A35B-Instruct; curated for SFT.
- Schema: `instance_id`, `repo`, `license`, `trajectory_id`, `trajectory`, `model_patch`, `dataset`.
- Format observed: OpenAI-style messages with tool calls for bash/editor-like tools.
- Fit: high for repo/code-agent action policy; not compact and not Hermes-tool-native.
- Use: mine successful trajectories with license allowlist; map bash/editor calls to Hermes `terminal`/`read_file`/`patch`; reject long narration.

### 3. `SWE-Gym/OpenHands-SFT-Trajectories`

- Why relevant: small successful OpenHands SFT set.
- Schema: `messages`.
- Format observed: explicit computer interaction tools (`execute_bash`, `str_replace_editor`, `finish`).
- Fit: high-quality but small; good adapter/converter prototype source.
- Use: first import candidate because size is manageable and success-filtered.

### 4. `zake7749/Qwen-3.6-plus-agent-tool-calling-trajectory`

- Why relevant: Qwen 3.6 Plus generated multi-turn tool-calling trajectories with score/reward; same family as our base model.
- Schema: `id`, `domain`, `score`, `reward`, `num_turns`, `messages`, `tools`.
- Format observed: customer-service ToolScale tasks; assistant turns include `reasoning_content` and OpenAI-style `tool_calls`.
- Fit: good for generic tool-calling policy and score-filtering; weaker for repo/Hermes-specific workflow.
- Use: filter `reward > 0` or high `score`, compress reasoning into `SCRATCH<=32/96`, keep only tool-discipline examples.

### 5. `jkazdan/taubench_traces_training_data`

- Why relevant: clean OpenAI-style tool-call conversations with `assistant` tool calls and `tool` results.
- Schema: `messages` with `tool_calls`.
- Fit: good for multi-turn tool-call mechanics, verification-before-write, and confirmation discipline; domain-specific retail/airline, not coding.
- Use: small, easy conversion source for tool-call discipline/preference pairs.

## Secondary candidates

### `smolagents/hermes-function-calling-v1-formatted-code-agent`

- Hermes/function-calling branded and small.
- Uses `<code>...</code>` executable snippets and domain functions.
- Useful for function-call syntax pressure, but not close to Hermes Agent repo/terminal behavior.

### `AmanPriyanshu/tool-reasoning-sft-TOOLS-hermes_reasoning_tool_use-data-cleaned-rectified`

- Cleaned Hermes reasoning tool-use corpus.
- But system prompt explicitly encourages extremely long `<think>` reasoning.
- Use only if aggressively stripped/compressed; not a direct SFT source for v0.

### `nebius/SWE-agent-trajectories`

- Large SWE-agent trajectories with target labels and generated patches.
- Uses SWE-agent-specific editor/search commands rather than our tool surface.
- Potentially useful, but conversion cost is higher than OpenHands/TauBench/Qwen ToolScale.

## Recommendation

Do not dump any of these directly into v0. Best next import order:

1. Prototype converter on `SWE-Gym/OpenHands-SFT-Trajectories`.
2. Add Qwen ToolScale high-score rows from `zake7749/Qwen-3.6-plus-agent-tool-calling-trajectory` for generic tool policy.
3. Add a filtered slice from `open-thoughts/OpenThoughts-Agent-v1-SFT` for terminal command behavior.
4. Only then consider larger OpenHands/SWE-Zero mining.

All imports should pass the existing data-alignment gate: parseable action JSON, no raw secrets, train/eval disjointness, no `SCRATCH<=80`, and GPT-like teacher scratch capped/tagged rather than silently mixed.
